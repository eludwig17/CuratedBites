# Schema Evolution Summary

The nine entities I proposed in Part I Users, Restaurant, Cuisine, Diet, Menu, Review, Favorite, and the two junction tables, had all survived to the final build, and so did every relationship between them. What changed was the level of detail. The Part I write-up described rules in plain English ("rating has to be between 1–5", "price range between 1 and 4", "items on menu has to be a positive value"), and in Part II I had to actually translate those into SQL: `CHECK (Rating BETWEEN 1 AND 5)` on Review, `CHECK (PriceRange BETWEEN 1 AND 4)` on Restaurant, and `CHECK (Price > 0)` on Menu. A few small naming and integrity issues got cleaned up at the same time. The original `User` table had to be renamed to `Users` because `USER` is a reserved word in mySQL, the second column of the `RestaurantCuisine` junction was written as just `Cuisine` in the proposal and corrected to `CuisineID`, and the Review table's `date` column was renamed to `ReviewDate` to avoid clashing with the `DATE` type keyword. I also added unique rules that the proposal implied but didn't formally state: `UNIQUE` on `Username` and `Email` so accounts can't collide, `UNIQUE` on `Cuisine.Name` and `Diet.Name` so the lookup tables stay clean, and a composite `UNIQUE(UserID, RestaurantID)` on Favorite so a user can't favorite the same restaurant twice. Sensible defaults were filled in along the way `CreationDate` and `ReviewDate` default to `CURRENT_DATE`, and `ReviewRatingCounter` defaults to `0`.

The Part IV work was the only point where new database objects were introduced rather than existing ones being refined. Two indexes were added once I could see how the Flask routes actually queried the data: `idxReviewRestaurant` on `Review(RestaurantID)`, since `/api/reviews/restaurant/<id>` and the `RestaurantSummary` aggregation both filter or join on that column constantly, and `idxFavoriteUser` on `Favorite(UserID)` because the existing composite `UNIQUE(UserID, RestaurantID)` index only helps when both columns are filtered together `/api/users/<id>/favorites` filters on UserID alone so it needed its own index. I also added the `RestaurantSummary` view, which rolls up cuisines, review count, average rating, and favorite count per restaurant in one place so the same five table join doesn't have to be rewritten every time an endpoint needs an overview. Looking back, almost none of the changes came from rethinking the data model the Part I design was solid in terms of which tables should exist and how they connect. The changes came from the gap between describing constraints in English and enforcing them in SQL, and from performance and code reusability problems that only showed up once real routes were hitting the real sample data that was added to the database.
